    Agent Dispatcher — Technical Reference¶

    Dispatcher version: 3.9.1 Faraday server version: 5.19.0 Communication protocol: Socket.IO (v3.9.1) / WebSocket (legacy) Async framework: Python asyncio

    This document provides a deep technical reference for the Faraday Agent Dispatcher, covering its architecture, communication protocols, executor lifecycle, and data flow.

    Architecture Overview¶

    +------------------+       Socket.IO       +-------------------+
|                  | <-------------------> |                   |
|  Faraday Server  |    REST API (HTTPS)   | Agent Dispatcher  |
|  (Flask + Redis) | <-------------------> | (Python asyncio)  |
|                  |                       |                   |
+------------------+                       +--------+----------+
       |                                           |
       | bulk_create API                           | subprocess
       |                                           | (stdin/stdout/stderr)
       v                                           v
+------------------+                       +-------------------+
|   PostgreSQL     |                       |   Executor        |
|   (results)      |                       |   (nmap, nuclei,  |
+------------------+                       |    burp, etc.)    |
                                           +-------------------+

    The dispatcher is a single-process, async Python application. It:

    1. Maintains a persistent Socket.IO connection to the Faraday server
    2. Receives RUN commands from the server (triggered by UI, scheduler, or API)
    3. Spawns executor scripts as subprocesses
    4. Reads stdout (JSON data) and stderr (logs) concurrently
    5. POSTs each JSON line to the Faraday bulk_create endpoint
    6. Reports execution status back via Socket.IO

    Components¶

    Faraday Server¶

    The Faraday server (Flask application) manages agents through these modules:

    Module Location Purpose
    api/modules/agent.py Agent CRUD, token management, run endpoint Agent lifecycle
    api/modules/agent_execution.py Execution history tracking Read-only history
    api/modules/agent_auth_token.py Registration token generation Token management
    api/modules/agents_schedule.py Scheduler CRUD, manual trigger Cron scheduling
    WebSocket server (Socket.IO) /dispatcher namespace Real-time communication

    Agent Dispatcher¶

    The dispatcher (faraday_agent_dispatcher) is composed of these modules:

    Module File Purpose
    Dispatcher dispatcher_io.py Core class: registration, Socket.IO connection, executor management
    DispatcherNamespace dispatcher_io.py Socket.IO event handler (on_connect, on_run, on_disconnect)
    Executor executor.py Executor model: config validation, command resolution, dependency checks
    StdOutLineProcessor executor_helper.py Reads executor stdout, parses JSON, POSTs to bulk_create API
    StdErrLineProcessor executor_helper.py Reads executor stderr, logs to dispatcher output
    Config config.py YAML/INI configuration loading and validation
    CLI cli/main.py Click-based CLI entry point (run, config-wizard)
    Wizard cli/wizard.py Interactive configuration wizard

    Executors¶

    Executors are standalone scripts located at:

    faraday_agent_dispatcher/static/executors/official/

    Each official executor has a corresponding manifest JSON file in faraday_agent_parameters_types that defines: - Required environment variables (credentials) - Runtime parameters (targets, options) with types and mandatory flags - Command template with dependencies - Tool category metadata

    Communication Protocol¶

    REST API Endpoints¶

    The dispatcher uses these Faraday server REST endpoints:

    Endpoint Method Auth Purpose
    /_api/config GET None Connectivity check (health check)
    /_api/v3/agents POST Registration token Register new agent, returns agent token
    /_api/v3/agent_websocket_token POST Agent token Obtain Socket.IO authentication token
    /_api/v3/ws/{workspace}/bulk_create POST Agent token Submit scan results (hosts, services, vulns)

    Authentication headers: - Registration: Authorization: Token <registration_token> (POST body) - Agent operations: Authorization: Agent <agent_token>

    Socket.IO Events¶

    The dispatcher connects to the /dispatcher Socket.IO namespace.

    Agent → Server events:

    Event Payload When
    join_agent {action: "JOIN_AGENT", token, executors: [{executor_name, args, category, tool}]} On connect
    run_status {action: "RUN_STATUS", execution_ids, executor_name, running, successful, message} During/after execution

    Server → Agent events:

    Event Payload When
    run {action: "RUN", execution_ids, workspaces, executor, args, plugin_args} Triggered by UI/scheduler/API

    WebSocket Messages (legacy)¶

    Older dispatcher versions used raw WebSocket connections to /websockets. The message format is similar but uses JSON.dumps over a raw WebSocket connection instead of Socket.IO events. The legacy code path still exists in dispatcher.py but the active path in v3.9.1 uses Socket.IO via dispatcher_io.py.

    Agent Registration Flow¶

    Agent Dispatcher                              Faraday Server
      |                                              |
      |--- GET /_api/config ----------------------->|  (1) Connection check
      |<-- 200 OK ----------------------------------|
      |                                              |
      |--- POST /_api/v3/agents ------------------->|  (2) Register agent
      |   {token, name, description}                 |      (one-time, with reg token)
      |<-- 200 {token: "<agent_token>"} ------------|
      |                                              |
      |   [agent_token saved to config YAML]         |
      |                                              |
      |--- POST /_api/v3/agent_websocket_token ----->|  (3) Get WS auth token
      |   Authorization: Agent <agent_token>         |
      |<-- 200 {token: "<ws_token>"} ---------------|
      |                                              |
      |=== Socket.IO connect /dispatcher ==========>|  (4) Establish connection
      |                                              |
      |--- emit("join_agent") --------------------->|  (5) Announce executors
      |   {action: "JOIN_AGENT",                     |
      |    token: ws_token,                          |
      |    executors: [...]}                         |
      |                                              |
      |   [Agent shows ONLINE in UI]                 |

    Error handling during registration:

    HTTP Status Meaning Action
    401 Invalid registration token Re-generate token in admin panel
    402 License expired/invalid Check Faraday license
    404 Server unreachable or wrong URL Verify host/port/SSL settings

    Faraday Server                  Dispatcher                      Executor Process
      |                              |                                |
      |--- emit("run") ------------>|                                |
      |   {executor, args,           |                                |
      |    workspaces,               |                                |
      |    execution_ids,            |                                |
      |    plugin_args}              |                                |
      |                              |                                |
      |                              |-- Validate parameters -------->|
      |                              |   (type check, mandatory check)|
      |                              |                                |
      |                              |-- Check dependencies --------->|
      |                              |   (verify tool is installed)    |
      |                              |                                |
      |<-- emit("run_status") ------|                                |
      |   {running: true}            |                                |
      |                              |-- create_subprocess_shell ---->|
      |                              |   (with env vars set)          |
      |                              |                                |
      |                              |       +------ stdout -------->|
      |                              |       |   (JSON lines)        |
      |                              |       +------ stderr -------->|
      |                              |       |   (log messages)      |

      |   (for each workspace)       |   parse JSON, add execution_id|

      |                              |                                |

    Environment Variable System¶

    Variable prefixes¶

    When the dispatcher spawns an executor subprocess, it sets environment variables from three sources:

    Source Prefix Set By Example
    Runtime arguments EXECUTOR_CONFIG_ Faraday server (from UI/API/scheduler) EXECUTOR_CONFIG_TARGET=192.168.1.0/24
    Plugin arguments AGENT_CONFIG_ Faraday server (shared filters/tags) AGENT_CONFIG_IGNORE_INFO=True
    Persistent credentials (no prefix) Config file varenvs section NESSUS_USERNAME=admin

    The executor reads these variables from its environment (e.g., os.environ["EXECUTOR_CONFIG_TARGET"] in Python, $EXECUTOR_CONFIG_TARGET in Bash).

    List parameters passed as EXECUTOR_CONFIG_* are JSON-encoded: EXECUTOR_CONFIG_TARGETS=["10.0.0.1","10.0.0.2"]

    List parameters passed as AGENT_CONFIG_* are comma-separated: AGENT_CONFIG_VULN_TAG=web,critical

    Common agent parameters¶

    These parameters are sent by the Faraday server as plugin arguments and are available to all executors:

    Variable Type Default Description
    AGENT_CONFIG_IGNORE_INFO Boolean False Skip informational-severity vulnerabilities
    AGENT_CONFIG_RESOLVE_HOSTNAME Boolean True Resolve hostnames to IP addresses
    AGENT_CONFIG_MIN_SEVERITY String (none) Minimum severity filter
    AGENT_CONFIG_MAX_SEVERITY String (none) Maximum severity filter
    AGENT_CONFIG_VULN_TAG CSV list (none) Tags to apply to vulnerabilities
    AGENT_CONFIG_SERVICE_TAG CSV list (none) Tags to apply to services
    AGENT_CONFIG_HOSTNAME_TAG CSV list (none) Tags to apply to hosts

    Parameter type system¶

    The faraday_agent_parameters_types package (v1.9.1) provides 10 validated types for executor parameters:

    Type Validation Example
    string Any non-empty string target_host
    integer Whole number port
    float Decimal number timeout
    boolean true / false verbose
    list JSON array of strings targets
    ip Valid IPv4/IPv6 address scan_target
    url Valid URL format api_endpoint
    password Masked string secret_key
    domains Comma-separated domain names target_domains
    range IP range (CIDR or dash notation) network_range

    The dispatcher validates parameters against their declared types before spawning the executor. Invalid parameters result in a RUN_STATUS error sent back to the server.

    Result Submission Pipeline¶

    Stdout processing¶

    The StdOutLineProcessor reads the executor's stdout line by line:

    1. Each line is expected to be a complete JSON object (one JSON document per line)
    2. The processor adds execution_id and command metadata to the JSON
    3. For each target workspace, the enriched JSON is POSTed to: 
      /_api/v3/ws/{workspace_name}/bulk_create
    4. Authentication uses the agent token: Authorization: Agent <token>
    5. Successful submission returns HTTP 201
    6. A 1-second delay is inserted between POSTs (rate limiting)

    Stderr processing¶

    The StdErrLineProcessor reads stderr line by line and logs each line to the dispatcher's output. Stderr output is informational and does not affect data submission.

    Bulk create payload format¶

    The executor's stdout must produce JSON matching the Faraday bulk_create schema:

    {
  "hosts": [
    {
      "ip": "192.168.1.1",
      "description": "Target host",
      "hostnames": ["server.example.com"],
      "os": "Linux",
      "services": [
        {
          "name": "http",
          "port": 80,
          "protocol": "tcp",
          "status": "open",
          "version": "Apache 2.4",
          "vulnerabilities": [
            {
              "name": "Apache Version Disclosure",
              "desc": "The server exposes its version number",
              "severity": "low",
              "refs": ["CVE-2024-XXXX"],
              "tags": ["web", "disclosure"],
              "data": "Additional technical details...",
              "type": "Vulnerability"
            }
          ]
        }
      ],
      "vulnerabilities": []
    }
  ]
}

    Command closure: When the executor's stdout stream ends (process exits), the dispatcher sends a final bulk_create with an empty hosts array and the command duration:

    {
  "hosts": [],
  "execution_id": 42,
  "command": {
    "tool": "production-scanner",
    "command": "nmap_scan",
    "user": "",
    "hostname": "",
    "params": "target=192.168.1.0/24",
    "import_source": "agent",
    "start_date": "2026-02-27T10:30:00",
    "duration": 45000000
  }
}

    The duration is in microseconds.

    Async Architecture¶

    The dispatcher uses Python asyncio with a single event loop. Running coroutines:

    1. Socket.IO client — Maintains the persistent connection to the server, dispatches events to the DispatcherNamespace handler
    2. on_run handler — Triggered per RUN command; validates parameters, spawns executor, manages subprocess lifecycle
    3. StdOutLineProcessor — Per-executor coroutine reading stdout
    4. StdErrLineProcessor — Per-executor coroutine reading stderr

    Multiple executors can run concurrently (each RUN spawns a new set of stdout/stderr reader coroutines).

    Graceful shutdown: On SIGINT or SIGTERM: 1. The dispatcher sends a LEAVE_AGENT message via WebSocket 2. All running executor tasks are cancelled 3. The Socket.IO connection is closed with code 1000 4. The process exits

    Configuration Internals¶

    Config file resolution order:

    1. Path specified via -c / --config-file CLI flag
    2. $FARADAY_HOME/config/dispatcher.yaml
    3. ~/.faraday/config/dispatcher.yaml

    Legacy INI migration: If a .ini file is found instead of .yaml, the dispatcher automatically converts it to YAML format, preserving all settings. The conversion handles: - Agent/executor section mapping - Single workspace → workspaces list migration - SSL defaults injection - Parameter type normalization

    Config validation (control_config): - Server section: host, ssl (boolean), ssl_cert (string, nullable), api_port (integer), websocket_port (integer) - Agent section: agent_name (string), executors (validated per-executor) - Tokens section: optional, validated if present

    Official Executor Manifest System¶

    Official executors are registered in faraday_agent_parameters_types with manifest files:

    {
  "cmd": "python {EXECUTOR_FILE_PATH}",
  "repo_executor": "nmap.py",
  "environment_variables": ["NMAP_EXTRA_ARGS"],
  "arguments": {
    "target": {
      "mandatory": true,
      "type": "ip",
      "base": "string"
    }
  },
  "check_cmds": ["nmap --version"],
  "category": ["Network & Vulnerability Scanners"]
}
    Manifest field Purpose
    cmd Command template. {EXECUTOR_FILE_PATH} is replaced with the absolute path to the executor script
    repo_executor Filename of the executor script in static/executors/official/
    environment_variables List of required env vars (credentials) that must be set in varenvs
    arguments Parameter definitions with type, mandatory flag, and base type
    check_cmds Shell commands to verify tool dependencies before execution
    category Tool category for UI grouping

    Custom Executor Development¶

    A custom executor is any script that:

    1. Reads configuration from environment variables
    2. Runs a security tool or data collection process
    3. Outputs Faraday-compatible JSON to stdout (one JSON document per line)
    4. Outputs logging/diagnostic info to stderr
    5. Exits with code 0 on success, non-zero on failure

    Minimal Python example:

    #!/usr/bin/env python3
import os
import json

target = os.environ.get("EXECUTOR_CONFIG_TARGET", "")

# Your tool logic here
results = {
    "hosts": [
        {
            "ip": target,
            "description": "Scanned host",
            "services": [],
            "vulnerabilities": []
        }
    ]
}

print(json.dumps(results))

    Minimal Bash example:

    #!/bin/bash
TARGET="${EXECUTOR_CONFIG_TARGET}"

# Run your tool and convert output to Faraday JSON
echo "{\"hosts\": [{\"ip\": \"${TARGET}\", \"services\": [], \"vulnerabilities\": []}]}"

    Register a custom executor in the config:

    agent:
  executors:
    my_tool:
      cmd: /path/to/my_executor.py
      max_size: 65536
      varenvs:
        API_KEY: "secret"
      params:
        target:
          mandatory: true
          type: string
          base: string

    Security Considerations¶

    • Agent tokens are stored in plaintext in the YAML config file. Restrict file permissions (chmod 600 dispatcher.yaml)
    • Credential env vars (in varenvs) are passed to executor subprocesses. Avoid logging them
    • SSL verification should be enabled in production. Use ssl_ignore: false and provide a valid ssl_cert path
    • The dispatcher uses asyncio.create_subprocess_shell to run executors. Executor commands are not user-supplied at runtime (they come from the config file), but ensure config files are not writable by untrusted users
    • Agent registration tokens are one-time use and should be treated as secrets
    • The HTTPS_PROXY environment variable is respected and automatically triggers SSL context adjustments
