    Agent Schedules API¶

    API Version: v3 (Faraday 5.10.0+) Base URL: https://localhost:5985/_api Authentication: Cookie-based session or API token (Authorization: Token <token>) Content-Type: application/json

    Agent Schedules allow you to automate recurring agent executions by defining cron-like timing, agent references, executor parameters, and workspace targets. Schedules run in the background and trigger the specified agent executor on the configured cadence.

    All endpoints are non-workspaced and operate under the /_api/v3/agents_schedule path prefix.

    AgentsSchedule Schema¶

    Property Type Required Description
    id integer read-only Primary key. Assigned by the server on creation.
    description string yes Human-readable description of the schedule (e.g., "Daily nmap scan").
    agent_id integer no ID of the agent to execute.
    executor string no Name of the executor to run (e.g., "nmap", "nikto").
    crontab string no Cron expression defining the schedule timing (e.g., "0 0 * * *" for daily at midnight).
    workspace_name string no Target workspace for the execution results.
    parameters object no Key-value pairs passed to the executor (e.g., {"target": "192.168.1.0/24"}).
    active boolean no Whether the schedule is currently enabled.

    Required fields for creation: description

    Note: The OpenAPI specification defines a minimal schema with only id and description. In practice, schedules are richer objects that include agent references, executor configuration, cron timing, and workspace targeting. The additional properties listed above are derived from the runtime behavior of the scheduling system.

    Example AgentsSchedule Object¶

    {
  "id": 1,
  "description": "Daily nmap scan",
  "agent_id": 5,
  "executor": "nmap",
  "crontab": "0 0 * * *",
  "workspace_name": "pentest",
  "parameters": {
    "target": "192.168.1.0/24"
  },
  "active": true
}

    Endpoints¶

    All endpoints operate under the non-workspaced path prefix /_api/v3/agents_schedule. No {workspace_name} path segment is required (the workspace is specified within the schedule object itself).

    1. List Schedules¶

    Retrieve a paginated list of all agent schedules.
    Method GET
    Path /_api/v3/agents_schedule
    Auth Session cookie or Authorization: Token <token>

    Query Parameters¶

    Parameter Type Default Description
    page integer 0 Page number (0-indexed).
    page_size integer 20 Number of results per page.
    sort string -- Field to sort by (e.g., id, description).
    sort_dir string asc Sort direction: asc or desc.

    Response: 200 OK¶

    {
  "rows": [
    {
      "id": 49,
      "key": 49,
      "value": {
        "description": "Monthly Scan *.site.com.ar",
        "cloud_agent_id": null,
        "service_tag": [],
        "executor": {
          "parameters_metadata": {
            "DOMAIN": {
              "base": "string",
              "type": "string",
              "mandatory": true
            }
          }

    Example¶

    curl -s -X GET \
  "https://localhost:5985/_api/v3/agents_schedule?page=0&page_size=10" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k

    Error Responses¶

    Code Condition
    401 Missing or invalid authentication.
    403 User lacks permission to view schedules.

    2. Create Schedule¶

    Create a new agent schedule.
    Method POST
    Path /_api/v3/agents_schedule
    Auth Session cookie or Authorization: Token <token>

    Request Body¶

    Field Type Required Description
    description string Yes Human-readable description of the schedule.
    agent_id integer No ID of the agent to execute.
    executor string No Name of the executor to invoke.
    crontab string No Cron expression for timing (e.g., "0 0 * * *").
    workspace_name string No Target workspace for execution results.
    parameters object No Key-value pairs to pass to the executor.

    Request Example¶

    {
  "description": "Daily nmap scan",
  "agent_id": 1,
  "executor": "nmap",
  "crontab": "0 0 * * *",
  "workspace_name": "pentest",
  "parameters": {
    "target": "192.168.1.0/24"
  }
}

    Response: 201 Created¶

    Returns the newly created schedule object with server-assigned fields.

    {
  "id": 4,
  "description": "Daily nmap scan",
  "agent_id": 1,
  "executor": "nmap",
  "crontab": "0 0 * * *",
  "workspace_name": "pentest",
  "parameters": {
    "target": "192.168.1.0/24"
  },
  "active": true
}

    Example¶

    curl -s -X POST \
  "https://localhost:5985/_api/v3/agents_schedule" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -H "Content-Type: application/json" \
  -k \
  -d '{
    "description": "Daily nmap scan",
    "agent_id": 1,
    "executor": "nmap",
    "crontab": "0 0 * * *",
    "workspace_name": "pentest",
    "parameters": {"target": "192.168.1.0/24"}
  }'

    Error Responses¶

    Code Condition
    400 Validation error (e.g., missing description, invalid crontab expression).
    401 Missing or invalid authentication.
    403 User lacks permission to create schedules.

    3. Get Schedule¶

    Retrieve a single schedule by its ID.
    Method GET
    Path /_api/v3/agents_schedule/{object_id}
    Auth Session cookie or Authorization: Token <token>

    Path Parameters¶

    Parameter Type Description
    object_id integer Schedule ID.

    Response: 200 OK¶

    Returns the full schedule object.

    {
  "id": 1,
  "description": "Daily nmap scan",
  "agent_id": 5,
  "executor": "nmap",
  "crontab": "0 0 * * *",
  "workspace_name": "pentest",
  "parameters": {
    "target": "192.168.1.0/24"
  },
  "active": true
}

    Example¶

    curl -s -X GET \
  "https://localhost:5985/_api/v3/agents_schedule/1" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k

    Error Responses¶

    Code Condition
    401 Missing or invalid authentication.
    403 User lacks permission to view this schedule.
    404 Schedule not found.

    4. Update Schedule (Full)¶

    Replace all mutable fields of a schedule. Fields not included in the request body are reset to their defaults.
    Method PUT
    Path /_api/v3/agents_schedule/{object_id}
    Auth Session cookie or Authorization: Token <token>

    Path Parameters¶

    Parameter Type Description
    object_id integer Schedule ID.

    Request Body¶

    Same schema as Create Schedule. The description field is required.

    Request Example¶

    {
  "description": "Nightly nmap full-port scan",
  "agent_id": 1,
  "executor": "nmap",
  "crontab": "0 1 * * *",
  "workspace_name": "pentest",
  "parameters": {
    "target": "192.168.1.0/24",
    "extra_args": "-p-"
  }
}

    Response: 200 OK¶

    Returns the updated schedule object.

    {
  "id": 1,
  "description": "Nightly nmap full-port scan",
  "agent_id": 1,
  "executor": "nmap",
  "crontab": "0 1 * * *",
  "workspace_name": "pentest",
  "parameters": {
    "target": "192.168.1.0/24",
    "extra_args": "-p-"
  },
  "active": true
}

    Example¶

    curl -s -X PUT \
  "https://localhost:5985/_api/v3/agents_schedule/1" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -H "Content-Type: application/json" \
  -k \
  -d '{
    "description": "Nightly nmap full-port scan",
    "agent_id": 1,
    "executor": "nmap",
    "crontab": "0 1 * * *",
    "workspace_name": "pentest",
    "parameters": {"target": "192.168.1.0/24", "extra_args": "-p-"}
  }'

    Error Responses¶

    Code Condition
    400 Validation error (e.g., missing description).
    401 Missing or invalid authentication.
    403 User lacks permission to update this schedule.
    404 Schedule not found.

    5. Update Schedule (Partial)¶

    Update only the specified fields of a schedule. Omitted fields remain unchanged.
    Method PATCH
    Path /_api/v3/agents_schedule/{object_id}
    Auth Session cookie or Authorization: Token <token>

    Path Parameters¶

    Parameter Type Description
    object_id integer Schedule ID.

    Request Body¶

    Only include the fields you want to change.

    Request Example¶

    {
  "crontab": "0 3 * * *",
  "description": "Updated: run at 3 AM"
}

    Response: 200 OK¶

    Returns the updated schedule object.

    Example¶

    curl -s -X PATCH \
  "https://localhost:5985/_api/v3/agents_schedule/1" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -H "Content-Type: application/json" \
  -k \
  -d '{"crontab": "0 3 * * *", "description": "Updated: run at 3 AM"}'

    Error Responses¶

    Code Condition
    400 Invalid field values.
    401 Missing or invalid authentication.
    403 User lacks permission to update this schedule.
    404 Schedule not found.

    6. Delete Schedule¶

    Delete a single schedule by its ID.
    Method DELETE
    Path /_api/v3/agents_schedule/{object_id}
    Auth Session cookie or Authorization: Token <token>

    Path Parameters¶

    Parameter Type Description
    object_id integer Schedule ID.

    Response: 204 No Content¶

    Empty response body on success.

    Example¶

    curl -s -X DELETE \
  "https://localhost:5985/_api/v3/agents_schedule/1" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k

    Error Responses¶

    Code Condition
    401 Missing or invalid authentication.
    403 User lacks permission to delete this schedule.
    404 Schedule not found.

    7. Run Schedule¶

    Manually trigger an immediate execution of a scheduled agent run, bypassing the cron timing.
    Method POST
    Path /_api/v3/agents_schedule/{schedule_id}/run
    Auth Session cookie or Authorization: Token <token>

    Path Parameters¶

    Parameter Type Description
    schedule_id integer Schedule ID to trigger.

    Request Body¶

    No request body is required. Send an empty body or {}.

    Response: 201 Created¶

    Returns confirmation that the scheduled run has been triggered.

    {
  "message": "Schedule triggered successfully"
}

    Example¶

    curl -s -X POST \
  "https://localhost:5985/_api/v3/agents_schedule/1/run" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -H "Content-Type: application/json" \
  -k \
  -d '{}'

    Error Responses¶

    Code Condition
    400 Schedule is misconfigured (e.g., missing agent or executor).
    401 Missing or invalid authentication.
    403 User lacks permission to trigger this schedule.
    404 Schedule not found.

    Common Error Codes¶

    All endpoints may return the following standard error responses:

    HTTP Status Meaning
    200 Success. Response body contains the requested data or confirmation.
    201 Created. New schedule was successfully created.
    204 No Content. Schedule was deleted successfully.
    400 Bad Request. Malformed JSON, missing required fields, invalid crontab expression, or invalid filter.
    401 Unauthorized. Authentication token is missing, expired, or invalid.
    403 Forbidden. Authenticated user does not have permission for the requested operation.
    404 Not Found. The schedule does not exist.
    500 Internal Server Error. Unexpected server-side failure.

    Error Response Format¶

    {
  "messages": ["Validation error: description is required"]
}

    Authentication¶

    All requests must include one of the following:

    API Token (recommended):

    Authorization: Token <your_api_token>

    Obtain a token from the Faraday web UI under Settings > API Tokens, or via:

    curl -s -X POST \
  "https://localhost:5985/_api/login" \
  -H "Content-Type: application/json" \
  -c cookies.txt \
  -k \
  -d '{"email": "faraday@example.com", "password": "changeme"}'

TOKEN=$(curl -s -X GET "https://localhost:5985/_api/v3/token" \
  -b cookies.txt -k | python3 -c "import sys,json; print(json.load(sys.stdin)['token'])")

    Session Cookie:

    Authenticate via the login endpoint and pass the session cookie with subsequent requests.

    Usage Notes¶

    Non-Workspaced Endpoints¶

    Unlike most Faraday API resources (hosts, services, vulnerabilities), agent schedules are not scoped to a workspace path segment. All schedule endpoints live under /_api/v3/agents_schedule. The target workspace for each schedule is specified within the schedule object's workspace_name field.

    Cron Expression Format¶

    The crontab field uses standard five-field cron syntax:

    * * * * *
| | | | |
| | | | +--- Day of week (0-6, Sun=0)
| | | +----- Month (1-12)
| | +------- Day of month (1-31)
| +--------- Hour (0-23)
+----------- Minute (0-59)

    Common examples:

    Expression Meaning
    0 0 * * * Every day at midnight.
    0 */6 * * * Every 6 hours.
    30 2 * * 1 Every Monday at 2:30 AM.
    0 0 * * 0 Every Sunday at midnight.
    */15 * * * * Every 15 minutes.

    Relationship to Agents and Executors¶

    • A schedule references an agent by agent_id and an executor by name (executor). The agent must be registered and active for the schedule to execute successfully.
    • The parameters object is passed directly to the executor at runtime. Valid keys depend on the specific executor (e.g., target, extra_args).
    • Use the Agents API to list available agents and their executors before creating schedules.

    Manual Triggers¶

    The /run endpoint allows you to test a schedule immediately without waiting for its cron timing. This is useful for verifying that the agent, executor, and parameters are configured correctly before relying on automated execution.

    Schedule Lifecycle¶

    1. Create a schedule with POST /_api/v3/agents_schedule.
    2. Verify the configuration by triggering a manual run with POST /_api/v3/agents_schedule/{id}/run.
    3. Monitor execution results via the Agent Executions API.
    4. Adjust timing or parameters with PATCH /_api/v3/agents_schedule/{id}.
    5. Remove schedules that are no longer needed with DELETE /_api/v3/agents_schedule/{id}.
