    Faraday Authentication & API Tokens — Complete Guide¶

    Faraday Server v5.10.0+ (Community & Commercial Editions)

    This guide covers every aspect of authenticating with the Faraday Security Platform: session management, API token lifecycle, agent tokens, SSO/LDAP integration, and security best practices. For the raw endpoint reference, see [[authentication-endpoints]].

    Authentication Methods Overview¶

    Faraday supports four authentication mechanisms, parsed from the Authorization HTTP header in app.py:load_user_from_request:

    Method Header Format Use Case CSRF Required
    Session Cookie Cookie: session=... Web UI, interactive scripts Yes (for writes)
    API Token (JWT) Authorization: Token <jwt> Automation, CI/CD, scripts No
    Basic Auth Authorization: Basic <base64> Quick one-off requests No
    Agent Token Authorization: Agent <token> Faraday Agent Dispatcher No

    All API endpoints require authentication by default. The default_login_required before-request handler in app.py rejects anonymous requests with HTTP 401 unless the endpoint is explicitly marked is_public = True.

    Session-Based Authentication¶

    How It Works¶

    1. Client sends credentials to POST /_api/login
    2. Server validates credentials via Flask-Security
    3. Server creates a server-side session and returns an HTTP-only session cookie
    4. Client includes the cookie in all subsequent requests
    5. For state-changing requests (POST, PUT, DELETE), a CSRF token is also required

    Login¶

    curl -k -s -X POST https://localhost:5985/_api/login \
  -H "Content-Type: application/json" \
  -c cookies.txt \
  -d '{"email": "faraday@example.com", "password": "changeme"}'

    Success (200): 

    {
  "response": {
    "user": {
      "authentication_token": null
    }
  }
}

    Session Verification¶

    Verify that a session is still valid and retrieve the CSRF token:

    curl -k -s -X GET https://localhost:5985/_api/session \
  -b cookies.txt

    Response: 

    {
  "csrf_token": "IjJhNmY4ZDk3ZTBk...",
  "preferences": { ... },
  "permissions": { ... },
  "user_id": 1
}

    The response includes:

    Field Type Description
    csrf_token string CSRF token for state-changing requests
    preferences object User's stored preferences
    permissions object Permission map (currently all-allowed)
    user_id integer Database ID of the authenticated user

    Idle Session Timeout¶

    Sessions expire after a configurable period of inactivity. Configure in ~/.faraday/config/server.ini:

    [faraday_server]
idle_session_timeout = 30

    Where the value is in minutes. Only server-hitting activity (clicking buttons, submitting forms, navigating pages) resets the timer. Simply viewing a page without interaction does not.

    Logout¶

    curl -k -s -X POST https://localhost:5985/_api/logout \
  -b cookies.txt

    After logout the session cookie is invalidated server-side. Discard the cookie file.

    API Token Authentication¶

    API tokens are the recommended method for automation, scripts, CI/CD pipelines, and any non-interactive use. They are stateless JWT tokens that do not require cookie management or CSRF tokens.

    Token Architecture¶

    Faraday API tokens are JSON Web Tokens (JWT) signed with HS512 (HMAC-SHA512). The JWT payload contains:

    Claim Description
    user_id User's fs_uniquifier (UUID from Flask-Security)
    validation_check Hash of the user's current password
    iat Issued-at timestamp (Unix epoch)
    exp Expiration timestamp (Unix epoch)

    The signing key is the Flask application's SECRET_KEY.

    Key design property: The validation_check claim contains a hash of the user's password at token-issuance time. If the user changes their password, the hash no longer matches and all previously issued tokens are immediately invalidated — without requiring a token blacklist or revocation list.

    Generating a Token¶

    Tokens are obtained via GET /_api/v3/token. This endpoint requires an existing authenticated session (cookie or Basic auth):

    # Step 1: Login
curl -k -s -X POST https://localhost:5985/_api/login \
  -H "Content-Type: application/json" \
  -c cookies.txt \
  -d '{"email": "faraday@example.com", "password": "changeme"}'

# Step 2: Get token
TOKEN=$(curl -k -s -X GET https://localhost:5985/_api/v3/token \
  -b cookies.txt)

echo "$TOKEN"

    Response (plain text — the raw JWT string): 

    eyJhbGciOiJIUzUxMiIsImlhdCI6MTcwOTAzMTU3OCwiZXhwIjoxNzA5MDc0Nzc4fQ...

    Alternatively, use Basic auth to skip the login step:

    TOKEN=$(curl -k -s --user "faraday@example.com:changeme" \
  https://localhost:5985/_api/v3/token)

    Audit trail: Every token request is logged to the audit log with the username, source IP, and timestamp.

    Using the Token¶

    Include the token in the Authorization header with the Token prefix:

    # List workspaces
curl -k -s -X GET https://localhost:5985/_api/v3/ws/ \
  -H "Authorization: Token $TOKEN"

# Create a vulnerability (no CSRF needed with token auth)
curl -k -s -X POST https://localhost:5985/_api/v3/ws/pentest/vulns/ \
  -H "Authorization: Token $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "SQL Injection",
    "severity": "critical",
    "parent": 42,
    "parent_type": "Host",
    "type": "Vulnerability"
  }'

    Python Example¶

    import requests

BASE = "https://localhost:5985"
session = requests.Session()
session.verify = False  # For self-signed certs

# Obtain token via Basic auth
resp = session.get(
    f"{BASE}/_api/v3/token",
    auth=("faraday@example.com", "changeme")
)
token = resp.text.strip('"')

# Use token for all subsequent requests
headers = {"Authorization": f"Token {token}"}
workspaces = session.get(f"{BASE}/_api/v3/ws/", headers=headers)
print(workspaces.json())

    Token Scoping & Permissions¶

    Current state (v5.10.0): API tokens carry the full permissions of the authenticated user. There is no per-token scope restriction. A token issued to an admin user grants admin-level access to all API endpoints.

    Persistent tokens (commercial feature): The database includes a UserToken model with a scope field supporting four values:

    Scope Purpose
    gitlab GitLab integration tokens
    scheduler Scheduled task tokens
    service_desk Service desk integration
    jira Jira integration tokens

    These scoped tokens also have alias (human-readable name), expires_at (optional expiration date), and revoked (boolean) fields. They are used by commercial-edition integrations but are not exposed through the community API.

    Basic Authentication¶

    Faraday supports HTTP Basic authentication for convenience. Credentials are sent base64-encoded in the Authorization header:

    curl -k -s --user "faraday@example.com:changeme" \
  https://localhost:5985/_api/v3/ws/

    This is equivalent to:

    curl -k -s -H "Authorization: Basic $(echo -n 'faraday@example.com:changeme' | base64)" \
  https://localhost:5985/_api/v3/ws/

    Caution: Basic auth sends credentials with every request. Prefer API tokens for production automation. Basic auth is best for quick one-off requests or initial token retrieval.

    Agent Tokens¶

    Agent tokens are a separate authentication mechanism used exclusively by the Faraday Agent Dispatcher for registering and communicating with the server.

    Agent Registration Token (TOTP)¶

    The agent registration token is a time-based one-time password (TOTP) generated using pyotp. It is used during the initial agent registration handshake.

    Endpoint: GET /_api/v3/agent_token

    curl -k -s -X GET https://localhost:5985/_api/v3/agent_token \
  -b cookies.txt

    Response: 

    {
  "token": "123456",
  "expires_in": 45,
  "total_duration": 60
}

    Field Description
    token Current TOTP code
    expires_in Seconds until this code expires
    total_duration Total TOTP interval (default: 60 seconds)

    Configuration:

    Setting Default Description
    agent_registration_secret Auto-generated TOTP secret key
    agent_token_expiration 60 (seconds) TOTP interval duration

    The agent_registration_secret is automatically generated on first server start and stored in ~/.faraday/config/server.ini.

    Persistent Agent Token¶

    Once an agent is registered, it receives a persistent 64-character random token stored in the agent database table. This token is used for ongoing agent-to-server communication:

    Authorization: Agent <64-char-token>

    Agent tokens are validated by direct database lookup (not JWT). If the agent token is not found, the server returns 403 Forbidden.

    Agent WebSocket Token¶

    For real-time communication, agents obtain short-lived WebSocket tokens:

    curl -k -s -X POST https://localhost:5985/_api/v3/agent_websocket_token \
  -H "Content-Type: application/json" \
  -d '{"token": "<agent-persistent-token>"}'

    The returned token is signed with itsdangerous.TimestampSigner using a websocket_agent salt and has a maximum age of 60 seconds.

    WebSocket Authentication¶

    Faraday uses WebSocket connections for real-time updates (new vulnerabilities, host changes, command output, etc.). WebSocket tokens are workspace-scoped.

    Obtaining a WebSocket Token¶

    curl -k -s -X GET \
  https://localhost:5985/_api/v3/ws/my_workspace/websocket_token \
  -b cookies.txt

    Response: 

    {
  "token": "d3Nfd29ya3NwYWNlX3Rva2VuX2V4YW1wbGU..."
}

    Connecting to WebSocket¶

    wss://localhost:5985/_api/v3/ws/my_workspace/websocket?token=<token>

    User WebSocket tokens are signed with itsdangerous.TimestampSigner using a websocket salt. The token encodes the workspace ID and is validated on connection.

    Requirements: - The user must have at least read access to the target workspace - The token is short-lived; obtain a fresh token before each connection

    User Roles & Permissions¶

    Faraday has four built-in roles:

    Role Description
    admin Full system access, user management
    pentester Standard penetration tester access
    asset_owner Asset/infrastructure owner view
    client Read-only client reporting access

    Current state (v5.10.0): The permission system in api/base.py currently returns "allowed" for all permissions for all authenticated users. The RBAC infrastructure is in place (roles, permission checks) but enforcement is handled at the application layer rather than per-endpoint. All authenticated users effectively have the same API access level regardless of role.

    Role assignments affect the web UI experience (menu items, available actions) more than API-level access control.

    Token Configuration & Expiration¶

    API Token Expiration¶

    The API token lifetime is configured via api_token_expiration in ~/.faraday/config/server.ini:

    [faraday_server]
api_token_expiration = 43200

    The value is in seconds. The effective default depends on your deployment:

    Deployment Method Default Value Duration
    Docker 604800 7 days
    Bare-metal (server.ini) 43200 12 hours
    Python fallback (no config) 86400 24 hours

    Important: These three different defaults can cause confusion. Check your actual server.ini to confirm which value is in effect.

    Renewing Expired Tokens¶

    Tokens cannot be refreshed. When a token expires, the client must:

    1. Re-authenticate (login or Basic auth)
    2. Request a new token from GET /_api/v3/token

    Password-Based Invalidation¶

    Changing a user's password immediately invalidates all outstanding tokens for that user. This happens because the JWT contains a validation_check claim with a hash of the password at issuance time. On each request, the server re-hashes the current password and compares it to the claim. If they differ, the token is rejected with 401.

    This is a security feature: if a token is compromised, changing the password revokes all tokens without needing a token blacklist.

    Token Revocation¶

    Automatic Revocation¶

    Trigger What Gets Revoked Mechanism
    Password change All JWT tokens for that user Password hash mismatch in JWT validation_check claim
    Token expiration The expired token JWT exp claim checked on every request
    Session logout The session cookie only Server-side session invalidation

    Manual Revocation¶

    There is no explicit token revocation endpoint in the community edition. To revoke all tokens for a user:

    1. Change the user's password — this immediately invalidates all outstanding JWTs 

      # Via API (as the user)
curl -k -s -X POST https://localhost:5985/_api/change \
  -H "Content-Type: application/json" \
  -b cookies.txt \
  -d '{
    "password": "old_password",
    "new_password": "new_password",
    "new_password_confirm": "new_password"
  }'

      # Via CLI (as admin)
faraday-manage change-password

    2. Rotate the server SECRET_KEY — this invalidates ALL tokens for ALL users (nuclear option, requires server restart)

    Commercial Persistent Tokens¶

    Commercial-edition UserToken records have a revoked boolean field. Setting revoked = True disables the token. These tokens also support expires_at for time-based expiration independent of the JWT mechanism.

    SSO & LDAP Integration¶

    LDAP¶

    LDAP authentication is a commercial feature that replaces local user authentication with Active Directory / LDAP directory validation.

    Key constraints: - Faraday does not support hybrid LDAP + local users — you must choose one - When use_local_roles = True, any AD user can access Faraday - Server restart is required after configuration changes

    Configuration via CLI: 

    faraday-manage settings -a update ldap

    LDAP settings:

    Setting Default Description
    enabled False Enable LDAP authentication
    server (empty) LDAP server hostname
    port 389 LDAP server port
    use_ldaps False Use LDAPS (port 636)
    use_start_tls False Use STARTTLS on plain port
    domain (empty) AD domain name
    domain_dn (empty) Domain distinguished name
    bind_format CN Bind DN format (CN or DN)
    bind_dn (empty) Bind distinguished name
    user_class user LDAP user object class
    user_attribute sAMAccountName Login attribute
    group_class group LDAP group object class
    paginated_fetch False Use paged results
    use_local_roles True Use Faraday role assignments (vs. LDAP groups)
    default_local_role admin Default role for new LDAP users
    admin_group (empty) LDAP group → admin role
    pentester_group (empty) LDAP group → pentester role
    asset_owner_group (empty) LDAP group → asset_owner role
    client_group (empty) LDAP group → client role
    disconnect_timeout 2.0 LDAP connection timeout (seconds)

    LDAP users and API tokens: LDAP-authenticated users can still obtain API tokens via GET /_api/v3/token after establishing a session. The token mechanism is independent of the authentication backend. However, since LDAP users may not have a local password stored, the validation_check claim behavior may differ.

    SAML¶

    SAML SSO is a commercial feature supporting single sign-on with identity providers such as Auth0, Azure AD, Okta, and Google Workspace.

    Key endpoints (SAML):

    Endpoint Purpose
    /_api/saml/login Initiate SAML login flow
    /_api/saml/acs Assertion Consumer Service (callback)
    /_api/saml/logout SAML single logout
    /_api/saml/metadata.xml SP metadata for IdP configuration

    Configuration via CLI: 

    faraday-manage settings -a update saml

    SAML settings:

    Setting Default Description
    enabled False Enable SAML authentication
    display_name Identity Provider Name shown on login page
    user_role admin Default role for SAML users
    admin_group (empty) SAML group → admin role
    attribute_identifier (empty) SAML attribute for username
    sso_url http://localhost/saml/login IdP login URL
    sp_certificate (empty) Service Provider certificate (PEM)
    sp_private_key (empty) Service Provider private key (PEM)
    idp_certificate (empty) Identity Provider certificate (PEM)

    Generating SP certificates: 

    openssl req -newkey rsa:2048 -nodes -keyout key.pem -x509 -days 1825 -out certificate.pem

    SAML + LDAP constraint: You cannot use both SAML and LDAP simultaneously. You can use SAML alongside local users.

    Supported identity providers (with docs): - Auth0 — see [[SAML-Auth0]] - Azure AD — see [[SAML-Azure]] - Okta — see [[SAML-Okta]] - Google Workspace — see [[SAML-Google]]

    CSRF Protection¶

    CSRF (Cross-Site Request Forgery) protection is enforced for session-based authentication only. Requests authenticated via API token, Basic auth, or Agent token bypass CSRF checks.

    When CSRF Is Required¶

    • Any POST, PUT, PATCH, or DELETE request using session cookies
    • File uploads (report imports) using session cookies

    How to Use¶

    1. Obtain the CSRF token from GET /_api/session: 

      CSRF=$(curl -k -s -X GET https://localhost:5985/_api/session \
  -b cookies.txt | python3 -c "import sys,json; print(json.load(sys.stdin)['csrf_token'])")

    2. Include it in state-changing requests via the X-CSRFToken header: 

      curl -k -s -X POST https://localhost:5985/_api/v3/ws/ \
  -H "Content-Type: application/json" \
  -H "X-CSRFToken: $CSRF" \
  -b cookies.txt \
  -d '{"name": "new_workspace"}'

    Bypassing CSRF¶

    Switch to API token authentication. Token-authenticated requests do not require CSRF tokens:

    curl -k -s -X POST https://localhost:5985/_api/v3/ws/ \
  -H "Content-Type: application/json" \
  -H "Authorization: Token $TOKEN" \
  -d '{"name": "new_workspace"}'

    Security Best Practices¶

    Token Management¶

    1. Prefer API tokens over Basic auth for production automation — tokens expire, passwords don't
    2. Store tokens securely — tokens grant full user-level access; treat them like passwords
    3. Set appropriate expiration — shorter lifetimes reduce the window of exposure for compromised tokens
    4. Rotate tokens periodically — generate new tokens regularly and discard old ones
    5. Use environment variables for tokens in scripts, never hardcode them

    Password Security¶

    1. Use strong passwords — Faraday enforces minimum complexity via Flask-Security configuration
    2. Change passwords to revoke tokens — this is the fastest way to invalidate all outstanding tokens
    3. Use faraday-manage change-password for administrative password resets

    Network Security¶

    1. Always use HTTPS — API tokens and credentials are sent in headers/bodies
    2. Deploy behind NGINX — use faraday-manage generate-nginx-config for a properly configured reverse proxy
    3. Restrict network access — limit API access to trusted networks where possible
    4. Configure idle session timeout — prevent unattended sessions from being hijacked

    SSO Recommendations¶

    1. Prefer SAML over LDAP when possible — SAML supports federated identity and allows keeping local users as a fallback
    2. Map LDAP/SAML groups to roles — avoid use_local_roles = True with empty group mappings (allows any AD user to access Faraday)
    3. Generate unique SP certificates per Faraday instance
    4. Restart the server after any SSO configuration changes

    Troubleshooting¶

    Common Issues¶

    Symptom Cause Solution
    401 Unauthorized on every request Token expired Re-authenticate and request new token
    401 after password change Token invalidated by password change Request new token
    403 Forbidden with session cookie Missing CSRF token Get CSRF from GET /_api/session
    403 Forbidden with agent token Agent not found in database Re-register agent
    Password reset email not arriving SMTP not configured Configure SMTP via faraday-manage settings -a update smtp
    LDAP login fails Server not restarted after config change systemctl restart faraday-server
    Token expires faster/slower than expected Conflicting api_token_expiration values Check actual server.ini value

    Debugging Authentication¶

    1. Check server logs — authentication failures are logged with details: 

      tail -f ~/.faraday/logs/faraday-server.log

    2. Verify token contents (for debugging only — do not expose in production): 

      # Decode JWT payload (base64, not verified)
echo "$TOKEN" | cut -d. -f2 | base64 -d 2>/dev/null | python3 -m json.tool

    3. Test authentication with curl verbose mode: 

      curl -k -v -X GET https://localhost:5985/_api/v3/ws/ \
  -H "Authorization: Token $TOKEN"

    4. Check audit log for token issuance events: 

      grep "requested token" ~/.faraday/logs/faraday-server.log

    Quick Reference Card¶

    Complete Token-Based Workflow¶

    #!/bin/bash
# Faraday API Token Authentication — Quick Start

SERVER="https://localhost:5985"
USER="faraday@example.com"
PASS="changeme"

# 1. Obtain token (via Basic auth — single request)
TOKEN=$(curl -k -s --user "$USER:$PASS" "$SERVER/_api/v3/token")

# 2. Use token for all API calls
curl -k -s -H "Authorization: Token $TOKEN" "$SERVER/_api/v3/ws/"

# 3. Create a workspace
curl -k -s -X POST "$SERVER/_api/v3/ws/" \
  -H "Authorization: Token $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name": "automated_scan"}'

# 4. Upload a report
curl -k -s -X POST "$SERVER/_api/v3/ws/automated_scan/upload_report" \
  -H "Authorization: Token $TOKEN" \
  -F "file=@nmap_results.xml"

    Environment Variable Pattern¶

    # Store credentials securely
export FARADAY_URL="https://faraday.example.com"
export FARADAY_TOKEN="eyJhbGciOiJIUzUxMiI..."

# Use in scripts
alias fapi='curl -k -s -H "Authorization: Token $FARADAY_TOKEN"'
fapi "$FARADAY_URL/_api/v3/ws/"
