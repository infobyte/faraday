    Export & Upload API¶

    Base URL: https://localhost:5985 API Version: v3 Authentication: Cookie-based session or API token (Authorization: Token <token>) Content-Type: application/json (unless noted otherwise)

    The Export & Upload API provides endpoints for exporting workspace data and importing tool reports. These endpoints enable integration with external tools and workflows, allowing data to be extracted for offline analysis or compliance reporting, and scan results to be ingested from 120+ supported security tools.

    Endpoints¶

    1. Export Workspace as XML¶

    Export the entire contents of a workspace (hosts, services, vulnerabilities, commands) as a Faraday XML file. This is useful for creating backups, migrating data between Faraday instances, or offline analysis.
    Method GET
    Path /_api/v3/ws/{workspace_name}/export_data
    Auth Required
    Content-Type Response: application/xml

    Path Parameters:

    Parameter Type Description
    workspace_name string Workspace slug to export.

    Response: 200 OK

    The response body is an XML file download containing all workspace data. The Content-Disposition header is set to trigger a file download.

    Response Headers:

    Header Value
    Content-Type application/xml
    Content-Disposition attachment; filename="{workspace_name}.xml"

    Example:

    curl -s -X GET \
  "https://localhost:5985/_api/v3/ws/pentest_q1/export_data" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k \
  -o pentest_q1.xml

    Example XML Structure:

    <?xml version="1.0" encoding="UTF-8"?>
<faraday_export workspace="pentest_q1">
  <hosts>
    <host>
      <id>42</id>
      <ip>192.168.1.1</ip>
      <description>Web server</description>
      <os>Linux 5.15</os>
      <services>
        <service>
          <port>80</port>
          <protocol>tcp</protocol>
          <name>http</name>
          <status>open</status>
        </service>
      </services>
      <vulnerabilities>
        <vulnerability>
          <name>SQL Injection</name>
          <severity>critical</severity>
          <status>open</status>
        </vulnerability>
      </vulnerabilities>
    </host>
  </hosts>
  <commands>
    <command>
      <tool>nmap</tool>
      <command_string>nmap -sV 192.168.1.0/24</command_string>
    </command>
  </commands>
</faraday_export>

    Error Codes:

    Code Reason
    401 Missing or invalid authentication.
    403 User lacks permission for this workspace.
    404 Workspace not found.

    2. Upload Tool Report¶

    Upload a security tool report file for parsing and import into a workspace. Faraday's plugin engine automatically detects the tool format and extracts hosts, services, and vulnerabilities from the report. This endpoint supports 120+ plugin formats including Nmap, Nessus, Burp Suite, Qualys, OpenVAS, OWASP ZAP, Nuclei, and many more.
    Method POST
    Path /_api/v3/ws/{workspace_name}/upload_report
    Auth Required
    Content-Type multipart/form-data

    Path Parameters:

    Parameter Type Description
    workspace_name string Target workspace slug for the imported data.

    Form Fields:

    Field Type Required Description
    file file yes The report file to upload. Supported formats include XML, JSON, CSV, HTML, and tool-specific formats.

    Response: 201 Created

    {
  "command_id": 25
}

    The command_id identifies the Command record created for this import. Use it to track the import via the Commands API (GET /_api/v3/ws/{ws}/commands/{command_id}).

    Example (Nmap XML report):

    curl -s -X POST \
  "https://localhost:5985/_api/v3/ws/pentest_q1/upload_report" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k \
  -F "file=@/path/to/nmap_scan.xml"

    Example (Nessus report):

    curl -s -X POST \
  "https://localhost:5985/_api/v3/ws/pentest_q1/upload_report" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k \
  -F "file=@/path/to/nessus_report.nessus"

    Example (Burp Suite report):

    curl -s -X POST \
  "https://localhost:5985/_api/v3/ws/pentest_q1/upload_report" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k \
  -F "file=@/path/to/burp_report.xml"

    Supported Tool Formats (partial list):

    Category Tools
    Network Scanners Nmap, Masscan, Zmap
    Vulnerability Scanners Nessus, OpenVAS, Qualys, Nexpose, Retina
    Web Application Scanners Burp Suite, OWASP ZAP, Nikto, Acunetix, AppScan, w3af
    DAST/SAST Tools Checkmarx, Fortify, SonarQube, Veracode, Snyk
    Exploitation Frameworks Metasploit, Core Impact
    Reconnaissance Shodan, Fierce, DNSrecon, theHarvester
    Compliance Lynis, CIS-CAT
    Cloud Security ScoutSuite, Prowler
    Container Security Trivy, Docker Bench
    Custom Nuclei, SQLMap, WPScan, SSLyze, testssl.sh

    For the full list of 120+ supported plugins, see the Faraday Plugins documentation.

    Error Codes:

    Code Reason
    400 No file provided, unrecognized file format, or file parsing error.
    401 Missing or invalid authentication.
    403 User lacks permission for this workspace.
    404 Workspace not found.

    Notes:

    • The upload is processed synchronously. Large reports may take several seconds to parse.
    • If the plugin cannot identify the report format, a 400 error is returned with a message indicating the format was not recognized.
    • Duplicate hosts, services, and vulnerabilities are merged automatically using the same deduplication logic as the Bulk Create endpoint.

    3. Export Vulnerabilities as CSV (Global)¶

    Export vulnerabilities across all accessible workspaces as a CSV file. This is useful for generating compliance reports or importing vulnerability data into external tools like spreadsheets or SIEM systems.
    Method GET
    Path /_api/v3/vulns/export_csv
    Auth Required
    Content-Type Response: text/csv

    Query Parameters:

    Parameter Type Default Description
    q string (JSON) (none) Optional filter expression (same format as filter endpoints).

    Response: 200 OK

    The response body is a CSV file download containing vulnerability data.

    Response Headers:

    Header Value
    Content-Type text/csv
    Content-Disposition attachment; filename="vulnerabilities.csv"

    CSV Columns:

    Column Description
    id Vulnerability ID.
    name Vulnerability name/title.
    severity Severity level.
    status Current status (open, closed, re-opened, risk-accepted).
    description Full description text.
    resolution Remediation guidance.
    host_ip IP address of the associated host.
    host_os Operating system of the associated host.
    service_name Name of the associated service (if applicable).
    service_port Port of the associated service (if applicable).
    workspace Workspace name.
    references Semicolon-separated list of references.
    cwe Semicolon-separated list of CWE identifiers.
    cve Semicolon-separated list of CVE identifiers.
    date Creation date.

    Example:

    curl -s -X GET \
  "https://localhost:5985/_api/v3/vulns/export_csv" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k \
  -o all_vulnerabilities.csv

    Example (with filter):

    curl -s -X GET \
  "https://localhost:5985/_api/v3/vulns/export_csv?q=%7B%22filters%22%3A%5B%7B%22name%22%3A%22severity%22%2C%22op%22%3A%22eq%22%2C%22val%22%3A%22critical%22%7D%5D%7D" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k \
  -o critical_vulnerabilities.csv

    Error Codes:

    Code Reason
    400 Malformed filter expression.
    401 Missing or invalid authentication.

    4. Export Vulnerabilities as CSV (Workspaced)¶

    Export vulnerabilities from a specific workspace as a CSV file.
    Method GET
    Path /_api/v3/ws/{workspace_name}/vulns/export_csv
    Auth Required
    Content-Type Response: text/csv

    Path Parameters:

    Parameter Type Description
    workspace_name string Workspace slug to export from.

    Query Parameters:

    Parameter Type Default Description
    q string (JSON) (none) Optional filter expression (same format as filter endpoints).

    Response: 200 OK

    The response body is a CSV file download containing vulnerability data from the specified workspace. The CSV format and columns are identical to the global CSV export.

    Response Headers:

    Header Value
    Content-Type text/csv
    Content-Disposition attachment; filename="{workspace_name}_vulnerabilities.csv"

    Example:

    curl -s -X GET \
  "https://localhost:5985/_api/v3/ws/pentest_q1/vulns/export_csv" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k \
  -o pentest_q1_vulnerabilities.csv

    Example (with filter -- only high and critical):

    curl -s -X GET \
  "https://localhost:5985/_api/v3/ws/pentest_q1/vulns/export_csv?q=%7B%22filters%22%3A%5B%7B%22name%22%3A%22severity%22%2C%22op%22%3A%22in%22%2C%22val%22%3A%5B%22high%22%2C%22critical%22%5D%7D%5D%7D" \
  -H "Authorization: Token YOUR_API_TOKEN" \
  -k \
  -o pentest_q1_high_critical.csv

    Error Codes:

    Code Reason
    400 Malformed filter expression.
    401 Missing or invalid authentication.
    403 User lacks permission for this workspace.
    404 Workspace not found.

    Common Error Codes¶

    HTTP Status Meaning
    200 Success. Response body contains the exported data or import confirmation.
    400 Bad Request. Invalid file, unrecognized format, malformed filter, or missing required fields.
    401 Unauthorized. Authentication token is missing, expired, or invalid.
    403 Forbidden. Authenticated user does not have permission for the requested workspace.
    404 Not Found. The workspace does not exist.
    500 Internal Server Error. Unexpected server-side failure (e.g., report parsing crash).

    Notes¶

    • All curl examples include -k to skip TLS certificate verification for local development. Remove this flag in production environments with valid certificates.
    • The XML export (export_data) includes all hosts, services, vulnerabilities, and commands in the workspace. For large workspaces this can produce a very large file.
    • The report upload endpoint (upload_report) uses multipart/form-data encoding, not application/json. Ensure the file field name is file.
    • CSV exports include all vulnerabilities matching the optional filter. Without a filter, all vulnerabilities are exported.
    • CSV exports support the same filter syntax used by the GET .../vulns/filter endpoints. The filter expression must be URL-encoded when passed as the q query parameter.
    • The command_id returned by upload_report can be used to query the Commands API to check import status and see how many hosts, services, and vulnerabilities were created.
    • Report processing applies the same deduplication logic as the Bulk Create API: existing hosts are updated, duplicate services are merged, and vulnerability duplicates are handled based on name/description/host/service matching.
