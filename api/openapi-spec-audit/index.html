    OpenAPI/Swagger Specification Audit & Comparison¶

    Generated: 2026-02-27 Phase: 05 — API & Reference Documentation Applies to: Faraday v5.17.0 (codebase spec) vs v5.10.0 (published wiki spec) Latest release: v5.19.0

    Executive Summary¶

    The Faraday API has two divergent OpenAPI specs: the codebase version (5.17.0) and the published docs version (5.10.0). Neither is complete. The published spec is 7 minor versions behind and missing critical new features, while the codebase spec is missing enterprise/commercial endpoints. Additionally, the api.faradaysec.com domain is defunct (NXDOMAIN/ECONNREFUSED).

    6 POST /v3/ws/<ws>/comment_unique comments.py Deduplicated comment creation
    7 POST /v3/ws/<ws>/credential/import_csv credentials.py CSV credential import

    Note: The spec includes empty path definitions for /_api/login, /_api/logout, /_api/change, and /_api/verify (no HTTP methods defined), indicating they were placeholder entries. The Flask-Security auth endpoints should be fully documented.

    7. Recommended Actions¶

    7.1 Immediate — Update Published Spec¶

    1. Replace the wiki faraday_swagger.json with the 5.17.0 codebase spec
    2. Source: repos/faraday-gitlab/faraday/openapi/faraday_swagger.json
    3. Target: repos/faraday-wiki/docs/files/faraday_swagger.json

    4. This immediately brings the Swagger UI from 5.10.0 to 5.17.0

    5. Remove all references to api.faradaysec.com across documentation

    6. Domain is defunct (NXDOMAIN / ECONNREFUSED)

    7. Redirect users to docs.faradaysec.com/api-swagger/

    8. Update the wiki api-swagger.md page to note the spec version

    7.2 Short-Term — Enrich the Spec¶

    1. Add Flask-Security auth endpoints to the spec with proper request/response schemas
    2. Add the public agent registration endpoint (POST /v3/agents without /_api)
    3. Add credential/import_csv endpoint to the spec
    4. Mark deprecated endpoints (deprecated: true) for skipped routes

    7.3 Medium-Term — Automation¶

    1. Set up CI pipeline to auto-export and publish the spec on each release
    2. Reconcile enterprise vs community endpoints — consider maintaining separate spec files
    3. Update to 5.19.0 when spec is regenerated from the latest codebase
