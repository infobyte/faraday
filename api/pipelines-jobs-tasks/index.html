    Pipelines, Jobs & Tasks API¶

    API Version: v3 Spec Version: Faraday 5.10.0 Base URL: https://localhost:5985 Authentication: Cookie-based session or API token (Authorization: Token <token>) Content-Type: application/json

    The Pipelines, Jobs & Tasks API provides Faraday's workflow automation engine. It allows users to define automated actions that execute when vulnerability data matches specified conditions. The three resources form a hierarchy: Pipelines orchestrate ordered sequences of Jobs, each Job evaluates Conditions against a data model and, when matched, executes one or more Tasks (atomic actions such as updating a field).

    Workflow Concepts¶

    Faraday's workflow automation engine lets you define rules that automatically act on vulnerability data. The system follows a three-tier hierarchy:

    Pipeline
  |
  +-- Job 1 (conditions: severity == "critical" AND confirmed == false)
  |     +-- Task A: SET confirmed = true
  |     +-- Task B: SET status = "open"
  |
  +-- Job 2 (conditions: severity == "info")
        +-- Task C: SET status = "closed"

    Pipeline¶

    A Pipeline is the top-level container that groups and orders a sequence of Jobs. Pipelines can be enabled/disabled, run on demand, exported, and cloned. Each pipeline is scoped to a workspace and defines jobs_order to control execution sequence.

    Job¶

    A Job defines what to match and what to do. Each Job targets a specific data model (e.g., "vulnerability") and contains:

    • Conditions/Rules (rules_json): A structured condition tree that determines which objects the job applies to. Conditions can be nested using logical operators (AND, OR) and support comparison operators against object fields (e.g., severity == "critical").
    • Tasks (tasks_ids): An ordered list of Task references that execute when conditions match.

    Jobs can belong to multiple Pipelines. They track execution history via WorkflowExecution records.

    Task¶

    A Task is the smallest unit of work -- a single atomic action. Tasks specify a command (e.g., "UPDATE"), a target field, and a value to set. Tasks can reference standard fields or custom fields. A single Task can be reused across multiple Jobs.

    Condition¶

    A Condition represents one node in a rule tree. Conditions can be:

    • Leaf nodes: Compare a field against data using an operator (e.g., field: "severity", operator: "==", data: "critical")
    • Branch nodes: Logical groupings (AND, OR) with children arrays containing nested conditions

    The is_root flag marks the top-level condition for a Job.

    Execution Flow¶

    1. A Pipeline is triggered (manually via run or by an event).
    2. Jobs execute in the order defined by jobs_order.
    3. For each Job, the condition tree (rules_json) is evaluated against matching objects.
    4. For every object that satisfies the conditions, each Task in tasks_ids order is executed.
    5. Each execution is recorded as a WorkflowExecution with success/failure status and a message.

    Schemas¶

    Pipeline Schema¶

    The Pipeline object represents an ordered workflow containing one or more Jobs.

    Property Type Required Read-Only Nullable Description
    id integer -- Yes No Unique pipeline identifier (auto-generated)
    name string -- No No Human-readable pipeline name
    description string -- No Yes Free-text description of the pipeline purpose
    jobs_order array -- No No Ordered array defining Job execution sequence
    enabled boolean -- No No Whether the pipeline is active and can be triggered
    workspace_id integer -- No No ID of the workspace this pipeline is scoped to
    jobs_ids array of integers -- No No List of Job IDs included in this pipeline
    running boolean -- Yes No Whether the pipeline is currently executing
    create_date string (date-time) -- Yes No ISO 8601 timestamp of creation
    update_date string (date-time) -- Yes No ISO 8601 timestamp of last modification

    Example Pipeline Object¶

    {
  "id": 1,
  "name": "Auto-triage critical vulns",
  "description": "Automatically confirm and prioritize critical vulnerabilities",
  "jobs_order": [1, 2],
  "enabled": true,
  "workspace_id": 1,
  "jobs_ids": [1, 2],
  "running": false,
  "create_date": "2026-02-15T10:00:00+00:00",
  "update_date": "2026-02-20T14:30:00+00:00"
}

    Job Schema¶

    The Job object defines a set of conditions and the tasks to execute when those conditions match.

    Property Type Required Read-Only Nullable Description
    id integer -- Yes No Unique job identifier (auto-generated)
    name string -- No No Human-readable job name
    description string -- No Yes Free-text description of the job purpose
    model string Yes No No The resource type this job targets (e.g., "vulnerability", "host", "service")
    enabled boolean -- No No Whether the job is active
    rules object -- No Yes Legacy rules object (use rules_json instead)
    rules_json object Yes No No Structured condition tree defining when this job triggers (see Condition Schema)
    tasks array -- Yes No Array of Task objects associated with this job (read-only, populated from tasks_ids)
    pipelines array -- Yes No Array of Pipeline objects that include this job
    tasks_ids array of integers Yes No No Ordered list of Task IDs to execute when conditions match
    metadata object -- No Yes Arbitrary metadata key-value pairs
    create_date string (date-time) -- Yes No ISO 8601 timestamp of creation
    update_date string (date-time) -- Yes No ISO 8601 timestamp of last modification

    Example Job Object¶

    {
  "id": 1,
  "name": "Auto-confirm critical",
  "description": "Automatically confirm vulnerabilities with critical severity",
  "model": "vulnerability",
  "enabled": true,
  "rules": {},

    "type": "AND",
    "is_root": true,
    "children": [
      {
        "type": "leaf",
        "field": "severity",
        "operator": "==",
        "data": "critical"
      }
    ]
  },
  "tasks": [
    {
      "id": 1,
      "name": "Set confirmed",
      "command": "UPDATE",
      "field": "confirmed",
      "value": "true"
    }
  ],
  "pipelines": [
    {
      "id": 1,
      "name": "Auto-triage critical vulns"
    }
  ],
  "tasks_ids": [1],
  "metadata": {},
  "create_date": "2026-02-15T10:05:00+00:00",
  "update_date": "2026-02-20T14:30:00+00:00"
}

    Task Schema¶

    The Task (Workflow Task) object defines a single atomic action to perform on a matched object.

    Property Type Required Read-Only Nullable Description
    id integer -- Yes No Unique task identifier (auto-generated)
    name string -- No No Human-readable task name
    description string -- No Yes Free-text description of the task
    command string Yes No No The action command to execute (e.g., "UPDATE", "DELETE", "ALERT")
    field string -- No Yes Target field name to act upon (e.g., "confirmed", "severity", "status")
    value string -- No Yes Value to set on the target field
    custom_field string -- No Yes Name of a custom field to act upon (mutually exclusive with field)
    job_id integer -- No Yes ID of the owning Job (if directly associated)
    target string -- No Yes Target scope or qualifier for the action
    create_date string (date-time) -- Yes No ISO 8601 timestamp of creation
    update_date string (date-time) -- Yes No ISO 8601 timestamp of last modification

    {
  "id": 1,
  "name": "Set confirmed",
  "description": "Mark vulnerability as confirmed",
  "command": "UPDATE",

  "update_date": "2026-02-15T10:03:00+00:00"
}

    WorkflowExecution Schema¶

    The WorkflowExecution object records the result of a single Job execution against a specific object.

    Property Type Required Read-Only Nullable Description
    id integer -- Yes No Unique execution record identifier
    successful boolean Yes No No Whether the execution completed successfully
    message string Yes No No Human-readable result message or error description
    object_and_id string Yes No No Identifier of the target object (format: "<type>:<id>", e.g., "vulnerability:42")
    create_date string (date-time) -- Yes No ISO 8601 timestamp of execution
    update_date string (date-time) -- Yes No ISO 8601 timestamp of last update

    Example WorkflowExecution Object¶

    {
  "id": 100,
  "successful": true,
  "message": "Updated field 'confirmed' to 'true' on vulnerability 42",
  "object_and_id": "vulnerability:42",
  "create_date": "2026-02-20T14:31:00+00:00",
  "update_date": "2026-02-20T14:31:00+00:00"
}

    Condition Schema¶

    The Condition object represents a single node in a Job's rule tree.

    Property Type Required Read-Only Nullable Description
    id integer -- Yes No Unique condition identifier
    type string Yes No No Node type: "AND", "OR" (branch), or "leaf" (comparison)
    parent_id integer -- No Yes ID of the parent condition (null for root)
    children array -- No No Array of child Condition objects (for branch nodes)
    field string -- No Yes Object field to compare (leaf nodes only, e.g., "severity", "confirmed")
    operator string -- No Yes Comparison operator (leaf nodes only, e.g., "==", "!=", ">", "<", ">=", "<=", "in", "like")
    data string -- No Yes Value to compare against (leaf nodes only)
    job_id integer -- No Yes ID of the Job this condition belongs to
    is_root boolean -- No No Whether this is the top-level condition node for its Job
    create_date string (date-time) -- Yes No ISO 8601 timestamp of creation
    update_date string (date-time) -- Yes No ISO 8601 timestamp of last modification

    Example Condition Tree¶

    {
  "id": 1,
  "type": "AND",
  "is_root": true,
  "parent_id": null,
  "job_id": 1,
  "children": [
    {
      "id": 2,
      "type": "leaf",
      "field": "severity",
      "operator": "==",
      "data": "critical",
      "parent_id": 1,
      "children": [],
      "is_root": false
    },
    {
      "id": 3,
      "type": "leaf",
      "field": "confirmed",
      "operator": "==",
      "data": "false",
      "parent_id": 1,
      "children": [],
      "is_root": false
    }
  ]
}

    Parameter Type Required Description
    object_id integer Yes The pipeline ID

    Response¶

    • Status: 204 No Content
    • Body: Empty
