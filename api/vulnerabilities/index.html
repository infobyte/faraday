    Vulnerabilities API¶

    Base URL: https://<faraday-server>/_api API Version: v3 Authentication: Cookie-based session or API token (Authorization: Token <token>) Content-Type: application/json (unless noted otherwise)

    The Vulnerabilities API is the largest endpoint group in Faraday, providing full lifecycle management for both standard infrastructure vulnerabilities and web application vulnerabilities. It exposes two scopes: workspaced (full CRUD within a workspace) and global (cross-workspace read-only access plus attachments and export).

    Vulnerability Schema¶

    The Vulnerability schema represents a standard infrastructure vulnerability finding.

    Field Type Required Read-Only Write-Only Nullable Description
    id integer -- yes -- -- Unique vulnerability identifier
    _id integer -- yes -- -- Alias for id (legacy CouchDB compatibility)
    _rev string -- yes -- -- Revision string (legacy CouchDB compatibility)
    name string yes -- -- yes Vulnerability name (min 1 character)
    severity string yes -- -- -- One of: "critical", "high", "medium", "low", "informational", "unclassified"
    type string yes -- -- -- Must be "vulnerability" for standard vulns or "vulnerability_web" for web vulns
    status string -- -- -- -- One of: "open", "opened", "closed", "re-opened", "risk-accepted"
    confirmed boolean -- -- -- -- Whether the vulnerability has been confirmed
    desc string -- -- -- -- Detailed description (used for create/update)
    description string -- yes -- -- Description (read-only alias for desc in responses)
    data string -- -- -- yes Additional technical details or proof data
    resolution string -- -- -- yes Recommended remediation steps
    tool string -- -- -- -- Name of the tool/scanner that found this vulnerability
    command_id integer -- -- yes -- ID of the command that created this vulnerability
    service object -- yes -- -- Nested Service object (read-only in responses)
    target string -- yes -- -- Target host/IP
    date datetime -- yes -- -- Creation date
    update_user string -- yes -- -- Username that last modified this vulnerability
    run_date datetime -- -- -- -- Scanner run date
    easeofresolution string -- -- -- yes One of: "trivial", "simple", "moderate", "difficult", "infeasible", or null
    external_id string -- -- -- yes External reference ID (e.g., from a scanner)
    cve array[string] -- -- -- -- List of CVE identifiers (e.g., ["CVE-2021-44228"])
    cwe array[string] -- -- -- -- List of CWE identifiers (e.g., ["CWE-89"])
    cvss2 object -- -- -- -- CVSS v2 scoring object
    cvss3 object -- -- -- -- CVSS v3 scoring object
    vulnerability_duplicate_id integer -- yes -- yes ID of the duplicate vulnerability
    vulnerability_template_id integer -- yes -- yes ID of the vulnerability template
    duplicates_associated array -- yes -- -- Associated duplicate vulnerability references
    duplicates_main array -- yes -- -- Main duplicate vulnerability references
    refs array[Reference] -- -- -- -- External references
    tags array[string] -- -- -- -- User-assigned tags
    impact object -- -- -- -- Impact assessment object
    custom_fields object -- -- -- -- Custom field values (keys match custom field schema)
    policyviolations array[string] -- -- -- -- Policy violation identifiers
    owned boolean -- yes -- -- Whether the vulnerability has been exploited
    owner string -- yes -- -- Username of the creator
    issuetracker object -- -- -- -- Issue tracker integration data
    metadata object -- -- -- -- Additional metadata
    obj_id string -- yes -- -- Object identifier string
    risk string -- yes -- -- Calculated risk score
    host_os string -- yes -- -- Operating system of the affected host
    hostnames array -- yes -- -- Hostnames associated with the affected host
    reference_instances array -- yes -- -- Expanded reference objects
    _attachments object -- -- -- -- Attachment metadata

    Embedded Service Object¶

    When a vulnerability is associated with a service, the service field in responses contains:

    Field Type Description
    _id integer Service ID
    ports integer/array Port number(s)
    status string "open", "closed", or "filtered"
    protocol string Protocol (e.g., "tcp", "udp")
    name string Service name (e.g., "http", "ssh")
    version string Service version
    summary string Human-readable summary

    VulnerabilityWeb Schema¶

    The VulnerabilityWeb schema extends the base Vulnerability schema with web-application-specific fields.

    Required fields: name, severity, type (must be "vulnerability_web"), parent, parent_type

    Field Type Required Description
    method string -- HTTP method (e.g., "GET", "POST")
    path string -- URL path where the vulnerability was found
    website string -- Target website/hostname
    request string -- Full HTTP request that triggered the vulnerability
    response string -- Full HTTP response received
    status_code integer -- HTTP response status code (nullable)
    params string -- Affected parameters
    pname string -- Primary parameter name
    query string -- Query string
    owasp string -- OWASP classification
    parent integer yes Parent host or service ID
    parent_type string yes Parent object type (e.g., "Host", "Service")

    All fields from the base Vulnerability Schema are also present.

    Evidence Schema¶

    Attachments (evidence files) associated with vulnerabilities use this schema.

    Field Type Read-Only Description
    content_type string yes MIME type of the attachment
    data binary yes File content (returned on download)
    description string -- Description of the evidence file

    Reference Schema¶

    External references attached to a vulnerability.

    Field Type Description
    name string Reference identifier or URL (e.g., "https://nvd.nist.gov/vuln/detail/CVE-2021-44228")
    type string Reference type (e.g., "url", "cve", "other")

    Workspaced Endpoints¶

    All workspaced endpoints require a {workspace_name} path parameter and operate within the scope of that workspace. There are 18 endpoints in total.

    Base path: /_api/v3/ws/{workspace_name}/vulns

    1. List Vulnerabilities¶

    Retrieve a paginated list of all vulnerabilities in a workspace.

    Property Value
    Method GET
    Path /_api/v3/ws/{workspace_name}/vulns
    Auth Required (session cookie or API token)
    Tags VulnerabilityGeneric

    Path Parameters:

    Parameter Type Required Description
    workspace_name string yes Workspace name

    Query Parameters:

    Parameter Type Default Description
    page integer 0 Page number (0-indexed)
    page_size integer 20 Items per page
    sort string -- Field to sort by (e.g., severity, name, date)
    sort_dir string asc Sort direction: asc or desc

    Response: 200 OK

    {
  "count": 42,
  "vulnerabilities": [
    {
      "id": 1,
      "_id": 1,
      "name": "SQL Injection",
      "severity": "critical",
      "type": "vulnerability",
      "status": "open",
      "confirmed": true,
      "desc": "SQL injection in login form parameter 'username'",
      "description": "SQL injection in login form parameter 'username'",
      "resolution": "Use parameterized queries",
      "target": "192.168.1.10",
      "tool": "sqlmap",
      "date": "2026-02-20T14:30:00Z",
      "update_date": "2026-02-25T10:00:00Z",
      "cve": ["CVE-2024-12345"],
      "cwe": ["CWE-89"],
      "tags": ["sqli", "critical"],
      "refs": [
        {"name": "https://owasp.org/www-community/attacks/SQL_Injection", "type": "url"}
      ],
      "workspace_name": "pentest-2026",
      "owner": "admin",
      "service": {
        "_id": 5,
        "ports": 443,
        "status": "open",
        "protocol": "tcp",
        "name": "https",
        "version": "Apache 2.4.51",
        "summary": "443/tcp (https)"
      },
      "custom_fields": {},
      "_attachments": {},
      "_rev": ""
    }
  ]
}

    Example:

    # List all vulnerabilities in workspace "pentest-2026"
curl -s -b cookies.txt \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns?page=0&page_size=50&sort=severity&sort_dir=desc"

# With API token
curl -s -H "Authorization: Token $TOKEN" \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns"

    Error Codes:

    Code Description
    401 Unauthorized -- not authenticated
    403 Forbidden -- insufficient permissions for this workspace
    404 Workspace not found

    2. Create Vulnerability¶

    Create a new vulnerability (standard or web) in a workspace.

    Property Value
    Method POST
    Path /_api/v3/ws/{workspace_name}/vulns
    Auth Required
    Tags VulnerabilityGeneric
    Request Body application/json -- Vulnerability or VulnerabilityWeb

    Path Parameters:

    Parameter Type Required Description
    workspace_name string yes Workspace name

    Request Body -- Standard Vulnerability:

    {
  "name": "SQL Injection",
  "severity": "critical",
  "type": "vulnerability",
  "desc": "SQL injection vulnerability found in login form parameter 'username'. An attacker can extract database contents.",
  "resolution": "Use parameterized queries or prepared statements for all database queries.",
  "confirmed": true,
  "data": "sqlmap identified the following injection point:\n  Parameter: username (POST)\n  Type: boolean-based blind",
  "tags": ["sqli"],
  "cve": ["CVE-2024-12345"],
  "cwe": ["CWE-89"],
  "refs": [
    {"name": "https://owasp.org/www-community/attacks/SQL_Injection", "type": "url"}
  ],
  "easeofresolution": "simple",
  "impact": {
    "accountability": true,
    "availability": false,
    "confidentiality": true,
    "integrity": true
  },
  "policyviolations": [],
  "custom_fields": {}
}

    Request Body -- Web Vulnerability:

    {
  "name": "Cross-Site Scripting (XSS)",
  "severity": "high",
  "type": "vulnerability_web",
  "desc": "Reflected XSS in search parameter allows script injection.",
  "resolution": "Sanitize and encode all user-supplied output.",
  "confirmed": true,
  "method": "GET",
  "path": "/search",
  "website": "https://target.example.com",
  "request": "GET /search?q=<script>alert(1)</script> HTTP/1.1\nHost: target.example.com",
  "response": "HTTP/1.1 200 OK\n\n<html>...<script>alert(1)</script>...</html>",
  "status_code": 200,
  "params": "q",
  "pname": "q",
  "query": "q=<script>alert(1)</script>",
  "parent": 42,
  "parent_type": "Host",
  "cve": [],
  "cwe": ["CWE-79"],
  "owasp": "A7:2017-Cross-Site Scripting (XSS)",
  "tags": ["xss", "reflected"]
}

    Response: 201 Created

    Returns the created vulnerability object with all fields populated (including id, date, owner, etc.).

    Example:

    # Create a standard vulnerability
curl -s -b cookies.txt -X POST \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "SQL Injection",
    "severity": "critical",
    "type": "vulnerability",
    "desc": "SQL injection in login form",
    "resolution": "Use parameterized queries",
    "confirmed": true,
    "tags": ["sqli"]
  }'

# Create a web vulnerability
curl -s -H "Authorization: Token $TOKEN" -X POST \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Reflected XSS",
    "severity": "high",
    "type": "vulnerability_web",
    "desc": "Reflected XSS in search parameter",
    "method": "GET",
    "path": "/search",
    "website": "https://target.example.com",
    "parent": 42,
    "parent_type": "Host"
  }'

    Error Codes:

    Code Description
    400 Validation error (missing required fields, invalid severity, etc.)
    401 Unauthorized
    403 Forbidden
    404 Workspace not found
    409 Conflict (duplicate vulnerability)

    3. Get Vulnerability¶

    Retrieve a single vulnerability by its ID.

    Property Value
    Method GET
    Path /_api/v3/ws/{workspace_name}/vulns/{object_id}
    Auth Required
    Tags VulnerabilityGeneric

    Path Parameters:

    Parameter Type Required Description
    workspace_name string yes Workspace name
    object_id integer yes Vulnerability ID

    Response: 200 OK

    Returns the full vulnerability object (same structure as list items).

    Example:

    curl -s -b cookies.txt \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns/1"

    Error Codes:

    Code Description
    401 Unauthorized
    404 Vulnerability or workspace not found

    4. Full Update Vulnerability¶

    Note: The PUT method on this path is not declared in the published 5.10.0 OpenAPI spec (only GET, PATCH, and DELETE are defined). It may exist at runtime via the standard Flask mixin but should be used with caution. Prefer PATCH for updates.

    Replace all fields of an existing vulnerability. All required fields must be provided.

    Property Value
    Method PUT
    Path /_api/v3/ws/{workspace_name}/vulns/{object_id}
    Auth Required
    Tags Vulnerability
    Request Body application/json -- full Vulnerability object

    Path Parameters:

    Parameter Type Required Description
    workspace_name string yes Workspace name
    object_id integer yes Vulnerability ID

    Request Body:

    All required fields (name, severity, type) must be included. Any omitted optional fields will be reset to their defaults.

    {
  "name": "SQL Injection (Updated)",
  "severity": "high",
  "type": "vulnerability",
  "desc": "Updated description with additional findings.",
  "resolution": "Use parameterized queries. Apply input validation.",
  "status": "risk-accepted",
  "confirmed": true,
  "tags": ["sqli", "accepted-risk"]
}

    Response: 200 OK

    Returns the updated vulnerability object.

    Example:

    curl -s -b cookies.txt -X PUT \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns/1" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "SQL Injection (Updated)",
    "severity": "high",
    "type": "vulnerability",
    "desc": "Updated description.",
    "status": "risk-accepted"
  }'

    Error Codes:

    Code Description
    400 Validation error
    401 Unauthorized
    404 Vulnerability or workspace not found

    5. Partial Update Vulnerability¶

    Update specific fields of an existing vulnerability without providing the full object.

    Property Value
    Method PATCH
    Path /_api/v3/ws/{workspace_name}/vulns/{object_id}
    Auth Required
    Tags VulnerabilityGeneric
    Request Body application/json -- partial Vulnerability or VulnerabilityWeb

    Path Parameters:

    Parameter Type Required Description
    workspace_name string yes Workspace name
    object_id integer yes Vulnerability ID

    Request Body:

    Only include the fields you want to change.

    {
  "severity": "medium",
  "status": "closed",
  "resolution": "Patched in version 2.1.0"
}

    Response: 200 OK

    Returns the updated vulnerability object.

    Example:

    # Close a vulnerability
curl -s -b cookies.txt -X PATCH \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns/1" \
  -H "Content-Type: application/json" \
  -d '{"status": "closed"}'

# Update severity and add a tag
curl -s -H "Authorization: Token $TOKEN" -X PATCH \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns/1" \
  -H "Content-Type: application/json" \
  -d '{"severity": "medium", "tags": ["reviewed"]}'

    Error Codes:

    Code Description
    400 Validation error
    401 Unauthorized
    404 Vulnerability or workspace not found
    409 Conflict

    8. Bulk Update Vulnerabilities¶

    Update one or more fields on multiple vulnerabilities in a single request.

    Property Value
    Method PATCH
    Path /_api/v3/ws/{workspace_name}/vulns
    Auth Required
    Tags VulnerabilityGeneric
    Request Body application/json

    Request Body:

    {
  "ids": [1, 2, 3],
  "update": {
    "severity": "high",
    "status": "closed"
  }
}

    The update object supports any writable field from the vulnerability schema.

    Response: 204 No Content

    Example:

    # Mark multiple vulnerabilities as closed
curl -s -b cookies.txt -X PATCH \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns" \
  -H "Content-Type: application/json" \
  -d '{"ids": [1, 2, 3], "update": {"status": "closed"}}'

# Bulk change severity
curl -s -H "Authorization: Token $TOKEN" -X PATCH \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns" \
  -H "Content-Type: application/json" \
  -d '{"ids": [10, 11, 12], "update": {"severity": "high", "confirmed": true}}'

    Error Codes:

    Code Description
    400 Invalid request body or update fields
    401 Unauthorized
    404 Workspace not found

    9. Count Vulnerabilities¶

    Get a count of vulnerabilities, optionally grouped by a field.

    Path Parameters:

    Parameter Type Required Description

    Query Parameters:

    Parameter Type Description

    Response: 200 OK

    {
  "total_count": 142,

    {"count": 23, "severity": "high"},
    {"count": 67, "severity": "medium"},
    {"count": 35, "severity": "low"},
    {"count": 12, "severity": "informational"}
  ]
}

    Example:

    # Total count
curl -s -b cookies.txt \

# Grouped by severity
curl -s -b cookies.txt \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns/count?group_by=severity"

# Grouped by status
curl -s -b cookies.txt \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns/count?group_by=status"

    Error Codes:

    Code Description
    401 Unauthorized
    404 Workspace not found

    11. Filter Vulnerabilities¶

    Retrieve vulnerabilities matching complex filter expressions.

    Property Value
    Method GET
    Path /_api/v3/ws/{workspace_name}/vulns/filter
    Auth Required
    Tags Filter, Vulnerability

    Query Parameters:

    Parameter Type Required Description
    q string yes JSON-encoded filter expression

    The filter expression uses the Faraday filter query language. See Faraday Filters documentation for full syntax.

    Common filter operators:

    Operator Description Example
    == Equals {"name": "severity", "op": "==", "val": "critical"}
    != Not equals {"name": "status", "op": "!=", "val": "closed"}
    has Contains (arrays) {"name": "tags", "op": "has", "val": "sqli"}
    like Pattern match {"name": "name", "op": "like", "val": "%injection%"}
    >, <, >=, <= Comparison {"name": "id", "op": ">", "val": 100}

    Filter expression structure:

    {
  "filters": [
    {"name": "severity", "op": "==", "val": "critical"}
  ]
}

    Compound filter (AND):

    {
  "filters": [
    {"name": "severity", "op": "==", "val": "critical"},
    {"name": "status", "op": "==", "val": "open"},
    {"name": "confirmed", "op": "==", "val": true}
  ]
}

    Response: 200 OK

    Returns a paginated list of matching vulnerabilities (same structure as List Vulnerabilities).

    Example:

    # Find all critical, open, confirmed vulnerabilities
curl -s -b cookies.txt \

    12. Upload Attachment¶

    Upload a file attachment (evidence) to a vulnerability.

    Property Value
    Method POST
    Path /_api/v3/ws/{workspace_name}/vulns/{vuln_id}/attachment

    Content-Type multipart/form-data

    Path Parameters:

    Parameter Type Required Description
    workspace_name string yes Workspace name
    vuln_id integer yes Vulnerability ID

    Form Fields:

    Field Type Required Description

    Response: 201 Created

    Example:

    # Upload a screenshot as evidence

  -F "description=SQL injection proof of concept"

# Upload a text file
curl -s -H "Authorization: Token $TOKEN" -X POST \
  "https://localhost:5985/_api/v3/ws/pentest-2026/vulns/1/attachment" \

    Global Endpoints¶

    Important: These global vulnerability endpoints are not declared in the published 5.10.0 OpenAPI specification (faraday_swagger.json). They exist in the codebase spec for version 5.17.0+. The paths and behaviors described below are derived from the codebase and may not be available in all Faraday deployments.

    Global endpoints provide cross-workspace access to vulnerabilities. They share the same request/response formats as their workspaced counterparts but operate without a workspace scope.

    Base path: /_api/v3/vulns

    G1. List Vulnerabilities (Global)¶

    Property Value
    Method GET
    Path /_api/v3/vulns
    Auth Required

    Retrieve vulnerabilities across all workspaces the authenticated user has access to. Supports the same pagination and sort parameters as the workspaced list endpoint.

    curl -s -b cookies.txt \
  "https://localhost:5985/_api/v3/vulns?page=0&page_size=100"

    G2. Get Vulnerability (Global)¶

    Property Value
    Method GET
    Path /_api/v3/vulns/{object_id}
    Auth Required

    Retrieve a single vulnerability by ID without specifying a workspace.

    curl -s -b cookies.txt \
  "https://localhost:5985/_api/v3/vulns/1"

    G3. Bulk Delete (Global)¶

    Property Value
    Method DELETE
    Path /_api/v3/vulns
    Auth Required
    Request Body application/json -- {"ids": [...]}

    Delete multiple vulnerabilities by IDs across workspaces.

    curl -s -b cookies.txt -X DELETE \
  "https://localhost:5985/_api/v3/vulns" \
  -H "Content-Type: application/json" \
  -d '{"ids": [1, 2, 3]}'

    Common Patterns¶

    Authentication¶

    All endpoints require authentication. Include one of the following with every request:

    # Option 1: Cookie-based session (obtain via POST /_api/login)
curl -b cookies.txt ...

# Option 2: API token
curl -H "Authorization: Token <base64-token>" ...

    Pagination¶

    All list endpoints support pagination:

    ?page=0&page_size=50&sort=severity&sort_dir=desc
    Parameter Type Default Description
    page integer 0 Page number (0-indexed)
    page_size integer 20 Results per page
    sort string -- Sort field
    sort_dir string asc asc or desc

    Severity Values¶

    Severity is a required field and must be one of:

    Value Description
    critical Critical severity
    high High severity
    medium Medium severity
    low Low severity
    informational Informational finding
    unclassified Not yet classified

    Status Values¶

    Value Description
    open Vulnerability is open and unresolved
    closed Vulnerability has been resolved
    re-opened Vulnerability was closed but detected again
    risk-accepted Vulnerability acknowledged but accepted as a risk

    Ease of Resolution Values¶

    Value Description
    trivial Can be fixed in minutes with no risk
    simple Straightforward fix, low risk
    moderate Requires planning, moderate effort
    difficult Significant effort, may involve architectural changes
    infeasible Cannot be practically fixed

    Vulnerability Type Differences¶

    Field vulnerability vulnerability_web
    Required fields name, severity, type name, severity, type, parent, parent_type
    Web-specific fields N/A method, path, website, request, response, status_code, params, pname, query, owasp
    Use case Infrastructure, network, OS-level findings Web application vulnerabilities (XSS, SQLi, CSRF, etc.)

    Standard Error Response¶

    All error responses follow this format:

    {
  "messages": ["Error description here"]
}

    Standard HTTP Response Codes¶

    Code Meaning Used By
    200 Success GET, PUT, PATCH (single), attachment operations
    201 Created POST (create vuln, upload attachment)
    204 No Content DELETE, PATCH (bulk operations)
    400 Bad Request Validation errors, invalid filter expressions
    401 Unauthorized Missing or invalid authentication
    403 Forbidden Insufficient permissions
    404 Not Found Workspace, vulnerability, or attachment not found
    409 Conflict Duplicate vulnerability
    500 Server Error Internal server error
